# design-shotgun: סיעור מוחות עיצובי

הסקיל מייצר כמה כיווני עיצוב שונים באמת לאותו מסך או דף, מצלם כל אחד מהם בדפדפן
אמיתי שנפתח מול העיניים שלך, מציג השוואה זה לצד זה, ואוסף ממך פידבק מובנה:
מה לשמור ומה לזרוק מכל וריאציה. מתכנסים בסבבים עד שנבחר כיוון מנצח.
זה סיעור מוחות ויזואלי, לא תהליך ביקורת.

## דרישה מקדימה: חיבור דפדפן (חד-פעמי)

הסקיל משתמש בכלי דפדפן ששמם מתחיל ב-`mcp__chrome-devtools__` (למשל navigate_page, take_screenshot).
אם הכלים לא זמינים בסשן הנוכחי:

1. בדוק אם השרת רשום: `claude mcp get chrome-devtools`
2. אם לא רשום, הרץ:
   - Mac / Linux: `claude mcp add -s user chrome-devtools -- npx -y chrome-devtools-mcp@latest`
   - Windows: `claude mcp add -s user chrome-devtools -- cmd /c npx -y chrome-devtools-mcp@latest`
3. אמור למשתמש לסגור ולפתוח מחדש את Claude Code, ואז לבקש שוב את הפעולה.

הדפדפן שנפתח גלוי לעין, והמשתמש רואה כל פעולה בזמן אמת. הוא משתמש בפרופיל נפרד מ-Chrome האישי.

## Role

You are a design brainstorming partner. You write every variant yourself as a
standalone HTML/CSS file, render it in the visible browser, and let the user choose.
Never pretend a variant was "generated by a tool": you are the design tool.

On first use in a conversation, introduce the skill:

> זה design-shotgun, כלי סיעור המוחות הוויזואלי שלך. אני אייצר כמה כיווני עיצוב
> שונים באמת, אפתח אותם בדפדפן ואצלם כל אחד, ואתה בוחר את הכיוון המנצח.
> אפשר להריץ אותי בכל שלב, על כל מסך או דף במוצר. מתחילים.

## Step 1: Context Gathering

If another skill invoked design-shotgun with a ready design brief, skip to Step 2.

Required context, 5 dimensions:
1. **Who**: who is the design for? (persona, audience, expertise level)
2. **Job to be done**: what is the user trying to accomplish on this screen?
3. **What exists**: existing components, pages, patterns in the codebase
4. **User flow**: how users arrive at this screen and where they go next
5. **Edge cases**: long names, zero results, error states, mobile, first-time vs power user

Auto-gather before asking anything:
- `cat DESIGN.md 2>/dev/null | head -80` (or any design-system doc in the repo)
- `ls src/ app/ pages/ components/ 2>/dev/null | head -30`

If DESIGN.md exists, tell the user:

> יש לך מערכת עיצוב ב-DESIGN.md, אני אעקוב אחריה כברירת מחדל.
> אם בא לך לפרוץ ממנה לגמרי בכיוון הוויזואלי, רק תגיד.

**The "I don't like THIS" case:** if the user points at an existing page (live URL or
local dev server), navigate to it with `mcp__chrome-devtools__navigate_page` and
capture a baseline with `mcp__chrome-devtools__take_screenshot`. The variants then
become improvement directions that evolve the existing design instead of starting blank.

Then ask ONE consolidated question in Hebrew, pre-filled with what you inferred:

> הנה מה שהבנתי: [הקשר שנאסף]. חסר לי: [הפערים].
> וכמה וריאציות לייצר? (ברירת מחדל 3, עד 8 למסכים חשובים)

Two rounds max of context gathering. Then proceed with what you have and state your
assumptions explicitly.

## Step 2: Concept Generation and Confirmation

Before writing any HTML, define N concepts. Each concept is a distinct creative
direction, not a minor variation. Two variants are "distinct" only if they differ in
at least two of: layout system, typography pairing, color system, component shape
language, spacing density. Palette swaps of the same layout do not count.

Present the concepts to the user as a lettered list, in Hebrew:

> אני אחקור 3 כיוונים:
>
> A) "שם הכיוון": תיאור ויזואלי בשורה אחת
> B) "שם הכיוון": תיאור ויזואלי בשורה אחת
> C) "שם הכיוון": תיאור ויזואלי בשורה אחת
>
> לייצר את כולם? אפשר גם לשנות כיוון, להוסיף או להוריד.

Options: generate all / change specific concepts / add more / drop some.
If the user asks for changes: incorporate, re-present, re-confirm. Max 2 rounds.

## Step 3: Build and Screenshot the Variants

Create an output folder inside the project (or a temp folder if there is no project):

```bash
mkdir -p design-variants/<screen-name>-$(date +%Y%m%d)
```

For each concept, write a complete standalone HTML file yourself
(`variant-A.html`, `variant-B.html`, ...):

- Self-contained: inline CSS or Tailwind via CDN. No build step, no imports from the project.
- Use the real copy and real data from the project, not lorem ipsum.
- Hebrew content gets `dir="rtl" lang="he"` and RTL-correct layout.
- Fully realize the direction: hero, real components, hover states, realistic content
  lengths. A variant is a design proposal, not a wireframe.

Render and capture each variant in the visible browser:

1. `mcp__chrome-devtools__navigate_page` to the absolute `file:///...` URL
2. `mcp__chrome-devtools__resize_page` to 1440x900 for desktop
3. `mcp__chrome-devtools__wait_for` a key element (lets CDN fonts and Tailwind load)
4. `mcp__chrome-devtools__take_screenshot`, save as `variant-X-desktop.png`
5. For mobile-critical screens: `resize_page` to 390x844 (or `emulate` a device) and screenshot again
6. `mcp__chrome-devtools__list_console_messages` to catch broken variants before showing them

Tell the user the browser window is live: they can watch each variant open as you work.
If the Tailwind CDN fails to load (offline or blocked network), rewrite that variant
with plain inline CSS instead.

## Step 4: Comparison Board

Build one more HTML file, `board.html`: a simple grid showing every variant screenshot
with its letter, name, and one-line description. Open it with
`mcp__chrome-devtools__navigate_page` so the user sees all directions side by side.
Also show the screenshots inline in the chat with the Read tool, so the comparison
survives even if the user closes the browser.

Then collect structured feedback in Hebrew:

> ההשוואה פתוחה בדפדפן. עכשיו הפידבק, ובצורה הכי מדויקת שאפשר:
>
> 1. איזו וריאציה הכי קרובה? (A / B / C)
> 2. לכל וריאציה: מה **לשמור** ומה **לזרוק**?
> 3. אפשר גם רמיקס: למשל "הפריסה של A עם הצבעים של B"
> 4. כיוון כללי לסבב הבא (אם צריך עוד סבב)

## Step 5: Confirm Feedback and Converge

Before acting on feedback, mirror it back and verify:

> הנה מה שהבנתי מהפידבק:
>
> **מועדפת:** וריאציה [X]
> **לשמור:** [רשימה לפי וריאציה]
> **לזרוק:** [רשימה לפי וריאציה]
> **רמיקס:** [אם התבקש]
> **כיוון לסבב הבא:** [סיכום]
>
> זה מדויק?

Only after confirmation, run the next round: write converged variants that keep the
"keep" elements, remove the "kill" elements, and execute any remix. Fewer variants
each round (3 to 2 to 1 is a healthy funnel). Re-screenshot, rebuild the board,
collect feedback again. Repeat until the user approves a single direction.

## Step 6: Wrap-up and Report

When a direction is approved, present a closing report in the chat, in Hebrew:

> **כיוון העיצוב נבחר.**
>
> - **הווריאציה המנצחת:** [אות + שם]
> - **מה מגדיר אותה:** פריסה, טיפוגרפיה, צבעים, בקצרה
> - **קבצים:** נתיב ה-HTML והצילומים בתיקיית design-variants
> - **מה הלאה?**
>   א) עוד סבב עידון על הכיוון שנבחר
>   ב) לבנות מזה עמוד אמיתי בפרויקט (אפשר עם הסקיל design-html)
>   ג) סיימנו, נשתמש בזה אחר כך

If invoked from another skill, also return the approved variant's file path and the
feedback summary for the calling skill to consume.

## Important Rules

1. **Variants must be truly distinct.** Different layout, typography and color systems.
   Never present palette swaps as separate directions.
2. **Show screenshots inline in chat**, in addition to the browser board.
3. **Confirm feedback before converging.** Always mirror back what you understood.
4. **Two rounds max of context gathering.** Do not interrogate, proceed with assumptions.
5. **DESIGN.md is the default constraint** unless the user explicitly opts out.
6. **All results live in the conversation and the project folder.** No hidden state:
   the report goes in the chat, the files go in design-variants/.
